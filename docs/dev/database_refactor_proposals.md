# Service/Database Interface Rework (V2)

The goal is to rework the service layer to be database-centric, removing the "media type category" abstraction (music, games, etc.) and allowing each database (RAWG, IGDB, MusicBrainz, etc.) to define its own entities, filters, sorting, and UI logic in a single interface.

## Proposed Changes

### Core Interface: `ExternalDatabase`

We will introduce a new interface that defines a database provider. This interface encapsulates all logic for a specific data source.

```typescript
export interface ExternalDatabase {
  id: string; // 'igdb', 'rawg', 'musicbrainz'
  label: string;
  icon: LucideIcon;
  entities: Record<string, DatabaseEntity>;
}

export interface DatabaseEntity {
  id: string; // 'game', 'artist', etc.
  label: string;
  labelPlural: string;
  icon: LucideIcon;
  colorClass: string;

  getSubtitle: (item: MediaItem) => string;
  getTertiaryText: (item: MediaItem) => string;

  // Search & Filters
  filters: FilterConfig[];
  sortOptions: SortOptionConfig[];
  defaultFilters: Record<string, unknown>;

  // Data Actions
  search: (query: string, options: SearchOptions) => Promise<SearchResult>;
  getDetails: (id: string) => Promise<MediaDetails>;
}
```

# Other proposed solution (2)

## Overview

As part of the V2 rework for the application's interfaces, we are shifting away from a global "Media Type" abstraction (e.g., generic "Music" or "Video Games" categories). Instead, we are adopting a **Database-First** approach.

In the previous architecture, different APIs (like RAWG and IGDB) were forced into a generic mold. This caused friction because their data structures, filtering capabilities, and search mechanics are fundamentally different.

The new architecture simplifies this by treating **each Database as the primary source of truth**. A **Database Provider** contains **Entities**, and each Entity encapsulates everything needed for the UI to render it, filter it, and fetch its data, with zero special logic required in the UI layer.

## Core Concepts

### 1. Standardized Item Format (`StandardItem` / `StandardDetails`)

Instead of defining generic domain models like `AlbumItem` or `GameItem` at the app level, all databases map their specific data into a highly generic `StandardItem` format. The UI expects this standard format and uses it directly.

- **Pre-computed Strings:** Fields like `subtitle` and `tertiaryText` are generated by the Database provider. The Board UI no longer needs to know how to format an item's data (e.g., it doesn't need to know that an album's subtitle should be its artist).
- **Unique Identification:** Items use a composite ID structure: `${databaseId}:${entityId}:${dbId}`. This guarantees global uniqueness and tells the app exactly where the item came from.
- **Extended Data:** `StandardDetails` includes an `extendedData` record, allowing databases to pass arbitrary specific data down to custom detail views without cluttering the global interface.

### 2. The Entity Interface (`DatabaseEntity`)

An Entity represents a specific type of data inside a database (e.g., a "Game" in RAWG, or a "Release Group" in MusicBrainz).

The Entity interface is the core of the new design. It defines exactly how the entity behaves, how to query it, what filters are available, and how to map its data.

- **Self-Contained UI Configuration:** Each entity provides its own `icon`, `colorClass`, and definitions for `filters` and `sortOptions`. The `SearchPanel` simply loops through these properties to build the search interface.
- **Encapsulated Data Fetching:** The `search` and `getDetails` methods live *inside* the Entity definition. They handle the API request **AND** the mapping to `StandardItem` or `StandardDetails`. There is no separate global mapper function.

### 3. The Database Provider (`DatabaseProvider`)

The top-level interface that the application registers. A Database Provider (e.g., `RAWGDatabase`) exports a list of `DatabaseEntity` objects.

## Benefits of the New Architecture

1. **Zero "Global" Logic:** The UI components (like `SearchPanel`) become completely generic. They just loop through `DatabaseProvider.entities` to create tabs, and loop through `entity.filters` to draw inputs. They don't need to know what a "Game" or "Album" is.
2. **Easy Extensibility:** Adding a new database (like OpenLibrary) simply requires creating an `OpenLibraryDatabase` object with its own `book` and `author` entities. No global mappers, type registries, or UI configurations need to be touched.
3. **No Mismatching:** Because `search` and `getDetails` live inside the Entity definition, it is impossible to accidentally send OpenLibrary-specific filters to TMDB. The typed `SearchParams` apply directly to the entity that defined the filters.
4. **Independent Evolution:** Databases can evolve independently. If RAWG adds a new filter for "VR Support", it only needs to be added to the RAWG `Game` entity definition. The UI will automatically render the new filter.

```ts
import { LucideIcon } from 'lucide-react';

/**
 * The standard item format that the Board and UI components expect.
 * Databases map their specific data into this generic format.
 */
export interface StandardItem {
  /** Globally unique app ID (e.g. `${databaseId}:${entityId}:${dbId}`) */
  id: string;
  /** The original ID in the source database */
  dbId: string;
  /** The identifier of the database provider (e.g., 'rawg', 'musicbrainz') */
  databaseId: string;
  /** The identifier of the entity type (e.g., 'game', 'album') */
  entityId: string;
  
  /** Primary display title */
  title: string;
  /** Optional image URL */
  imageUrl?: string;
  
  /** Pre-computed strings for the UI to avoid complex formatting logic in components */
  subtitle?: string; // e.g., "FromSoftware • 2022" or "Pink Floyd"
  tertiaryText?: string; // e.g., "PlayStation 5, PC, Xbox" or "Psychedelic Rock"
  
  /** Normalized rating (usually 0-10 or 0-100) */
  rating?: number;
}

/**
 * Deep metadata fetched on demand for an item.
 */
export interface StandardDetails extends StandardItem {
  /** A full description or biography */
  description?: string;
  /** Descriptive tags or genres */
  tags?: string[];
  /** External resource links (Wikipedia, Official Site, etc.) */
  externalLinks?: { label: string; url: string }[];
  /** Flexible record for any extra data specific to a database/entity */
  extendedData?: Record<string, unknown>;
}

/**
 * Standardized search response from any entity.
 */
export interface SearchResult {
  /** List of mapped standard items */
  items: StandardItem[];
  /** Total count of matches in the remote database */
  totalCount: number;
  /** Total number of pages available */
  totalPages: number;
  /** The current page index returned */
  currentPage: number;
}

// --- Search & Filtering ---

/**
 * The types of UI inputs supported for filters.
 */
export type FilterInputType = 
  | 'text' 
  | 'number' 
  | 'boolean' 
  | 'select' 
  | 'multiselect' 
  | 'range' 
  | 'date';

/**
 * Represents a single option for select/multiselect filters.
 */
export interface FilterOption {
  label: string;
  value: string | number;
  icon?: LucideIcon;
}

/**
 * Definition for a filter that the UI should render.
 */
export interface FilterDefinition {
  /** Unique ID for the filter (used as key in search params) */
  id: string;
  /** Human readable label for the UI */
  label: string;
  /** The type of input to render */
  type: FilterInputType;
  /** Available options for selection-based inputs */
  options?: FilterOption[];
  /** Default value for the filter state */
  defaultValue?: unknown;
  /** Placeholder text for inputs */
  placeholder?: string;
  /** Optional helper text shown to the user */
  helperText?: string;
}

/**
 * Definition for a sort option supported by the entity.
 */
export interface SortDefinition {
  /** Unique ID for the sort option */
  id: string;
  /** Human readable label for the UI (e.g. "Release Date") */
  label: string;
  /** Default direction if this sort is selected */
  defaultDirection?: 'asc' | 'desc';
}

/**
 * Parameters passed to the search function of an entity.
 */
export interface SearchParams {
  /** The raw text query */
  query: string;
  /** A map of filter values (ID -> Value) */
  filters: Record<string, unknown>;
  /** The ID of the selected sort option */
  sort?: string;
  /** The direction of the sort */
  sortDirection?: 'asc' | 'desc';
  /** The page to fetch (1-indexed) */
  page: number;
  /** The number of items to fetch per page */
  limit: number;
}

// --- Hierarchy ---

/**
 * Represents a specific type of data inside a database (e.g., "Game" in RAWG).
 * This interface encapsulates everything the UI needs to handle this entity.
 */
export interface DatabaseEntity {
  /** Unique entity ID (e.g., 'game', 'developer', 'album') */
  id: string;
  /** Singular name (e.g., 'Game') */
  label: string;
  /** Plural name (e.g., 'Games') */
  labelPlural: string;
  /** Icon component for the search tab and UI indicators */
  icon: LucideIcon;
  /** Tailwind color class for branding (e.g., 'bg-purple-500') */
  colorClass: string;

  /** Discovery configuration: what filters and sorts are available */
  filters: FilterDefinition[];
  sortOptions: SortDefinition[];
  
  /** 
   * Search method: Fetches and maps raw API data directly into StandardItems.
   */
  search: (params: SearchParams) => Promise<SearchResult>;
  
  /** 
   * Detail method: Fetches and maps deep metadata for a single item.
   */
  getDetails: (dbId: string) => Promise<StandardDetails>;
}

/**
 * Represents an independent external database service provider.
 * This is the top-level object registered in the application.
 */
export interface DatabaseProvider {
  /** Unique provider ID (e.g., 'rawg', 'igdb', 'musicbrainz') */
  id: string;
  /** Human readable name (e.g., 'RAWG Database') */
  label: string;
  /** Optional description of what this database provides */
  description?: string;
  /** Optional icon representing the service provider */
  icon?: LucideIcon;

  /** The list of entities this database exposes to the user */
  entities: DatabaseEntity[];
  
  /** Optional lifecycle hook for initialization (e.g., auth tokens) */
  initialize?: () => Promise<void>; 
}
```

## Goal
Rework the service/database interface to be self-describing and database-centric. This allows adding new databases (e.g., IGDB, RAWG) independently and declaratively, specifying their item types, filters, sorting, and UI mapping in a single place.

## Proposed Changes

### 1. New Core Interfaces

We will introduce a `Database` interface that encapsulates everything about a data source.

#### [NEW] [database.ts](file:///Users/paquerot/Perso/dev_projects/tierlist/lib/services/v2/database.ts)

```typescript
export interface Database {
  id: string; // e.g., 'rawg'
  label: string;
  icon: string; // Icon name from a whitelist or Lucide name
  
  // The types of entities this database provides
  entities: EntityDefinition[];
}

export interface EntityDefinition {
  id: string; // e.g., 'game'
  label: string;
  labelPlural: string;
  
  // UI mapping for the MediaCard
  ui: {
    colorClass: string;
    getSubtitle: (item: BaseItem) => string;
    getTertiaryText: (item: BaseItem) => string;
  };

  // Search configuration
  search: {
    // Declarative filter definitions
    filters: FilterField[];
    sortOptions: SortOption[];
    
    // Functional bits for the actual implementation
    // We keep these as functions to allow for API-specific logic while keeping the data description separate
    execute: (query: string, options: SearchOptions) => Promise<SearchResult>;
  };

  // Details configuration
  details: {
    execute: (id: string) => Promise<ItemDetails>;
  };
}

export interface FilterField {
  id: string;
  label: string;
  type: 'text' | 'select' | 'range' | 'toggle-group';
  options?: { label: string; value: string }[];
  
  /** 
   * Declarative mapping to API parameters.
   * If omitted, maps id directly to param name.
   */
  mapping?: {
    param: string;
    /** For 'range' type: how to format the values. Default: 'comma-separated' */
    format?: 'comma-separated' | 'min-max-params' | 'array';
    minParam?: string; // used if format is 'min-max-params'
    maxParam?: string; // used if format is 'min-max-params'
  };
}
```

### 2. Decoupling from Media Types
- Remove the `BoardCategory` (Music, Cinema, etc.) dependency from services.
- Move from a global `MediaType` enum to per-database `EntityID`.
- Standardize `BaseItem` to be more flexible, carrying a `metadata` record that UI components can selectively display.

### 3. Declarative Filter Application
Instead of hardcoding `if (filter.year) params.dates = ...`, we will provide a utility or a standard way in `EntityDefinition` to describe how filters map to API parameters.

### 4. Registry V2
The `MediaServiceRegistry` becomes a `DatabaseRegistry`. The UI (Search Panel) will now:
1. List available **Databases**.
2. For the selected database, list available **Entity Types**.
3. Render filters and results based on the `EntityDefinition`.
